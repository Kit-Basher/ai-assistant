# Bypass Behavior Map

This file tracks intentional pre-chat and Telegram-local paths that can return user-facing text before the normal `AgentRuntime.chat()` to `Orchestrator.handle_message()` path.

| Path | Trigger | File/function | Return boundary | Response shape | Intentional | Risk |
| --- | --- | --- | --- | --- | --- | --- |
| JSON request rejection | Invalid JSON, non-JSON content type, oversized body, invalid content length | `agent/api_server.py::APIServerHandler.do_POST()` via `_json_request_error_response()` | Before `/chat` | Failure envelope with `ok:false`, `error_kind:bad_request`, `message`, optional `next_question`, `recovery`, `envelope` | Yes. The body cannot be safely routed. | Low |
| Recovery chooser reply | Active legacy fix-it/recovery chooser and reply parses as a recovery option | `agent/api_server.py::APIServerHandler.do_POST()` and `AgentRuntime.consume_clarify_recovery_choice()` | Before `AgentRuntime.chat()` | Chat-style envelope with explicit `message`, `did_work`, optional `next_question` | Yes. Compatibility for operator recovery continuations. | Medium |
| Binary clarification reply | Active binary clarification state for source/user and reply is `1`/`2`, `A`/`B`, or option label | `agent/api_server.py::APIServerHandler.do_POST()` and `AgentRuntime.consume_binary_clarification_choice()` | Before `AgentRuntime.chat()` | Chat-style envelope, status answer or task-continuation prompt | Yes. It consumes a pending clarification exactly once. | Medium |
| Intent choice reply | Active intent choice state and reply is `chat`, `ask`, `model`, or numeric/letter alias | `agent/api_server.py::APIServerHandler.do_POST()` and `AgentRuntime.consume_intent_choice()` | Before `AgentRuntime.chat()` | Chat-style envelope with explicit next-step wording | Yes. It preserves old chooser flows. | Medium |
| Thread integrity resume | Active thread-integrity state and reply is same/new thread alias | `agent/api_server.py::APIServerHandler.do_POST()` and `AgentRuntime.consume_thread_integrity_choice()` | Before original `/chat`; resumes by calling `AgentRuntime.chat()` with sanitized resume payload | Normal chat body from resumed payload, or clarification expiry envelope | Yes. It is a continuation path that rewrites thread choice into a normal chat call. | Medium |
| Empty/low-confidence request | `/chat` or `/ask` has no explicit user message or low-confidence body | `agent/api_server.py::APIServerHandler._clarification_payload()` | Before `AgentRuntime.chat()` | `needs_clarification` envelope with one concrete question and hints | Yes. There is no useful request to route. | Low |
| Ambiguous binary prompt | Legacy allowed, no authoritative runtime route, ambiguity classifier fires | `agent/api_server.py::APIServerHandler._ambiguity_payload()` | Before `AgentRuntime.chat()` | `needs_clarification` envelope, either clarify text or setup suggestion text | Yes. Compatibility path; assistant frontdoor suppresses it for ordinary chat. | Medium |
| Intent ambiguity prompt | Legacy allowed and deterministic intent assessment asks to clarify | `agent/api_server.py::APIServerHandler._intent_ambiguity_payload()` | Before `AgentRuntime.chat()` | `needs_clarification` envelope with explicit question and intent assessment metadata in envelope | Yes. Compatibility path; assistant frontdoor suppresses it for ordinary chat. | Medium |
| `/start` and `/help` | Telegram text command `/start`, `start`, `/help`, `help`, `commands` | `agent/telegram_bridge.py::classify_telegram_text_command()` and `handle_telegram_command()` | Before API chat proxy and before Orchestrator | Command result with `text`, `route:help`, command metadata | Yes. Telegram affordance; no model needed. | Low |
| `/setup` and `/setup_token` | Telegram setup command or token rotation command | `agent/telegram_bridge.py::handle_telegram_command()` | Before API chat proxy and before Orchestrator | Setup guidance or token-rotation steps with `next_action` | Yes. Setup must work when chat is unavailable. | Medium |
| `/status` | Telegram status command aliases | `agent/telegram_bridge.py::build_telegram_status()` | Before API chat proxy and before Orchestrator | Runtime summary aligned to `/ready` and `/llm/status` fields | Yes. Health/status must work without chat. | Low |
| `/doctor` | Telegram `doctor`, `fix`, `diagnose`, `diagnostics`, `run doctor` | `agent/telegram_bridge.py::_build_telegram_doctor()` | Before API chat proxy and before Orchestrator | Doctor summary with pass/warn/fail counts and next command | Yes. Operator recovery shortcut. | Medium |
| `/health`, `/brief`, `/memory` command | Telegram command route with local orchestrator available | `agent/telegram_bridge.py::handle_telegram_command()` | Before API chat proxy, but uses Orchestrator command handling | Command result sanitized for Telegram | Yes. These are explicit local command shortcuts. | Medium |
| Memory-like text without bound runtime | Telegram text such as `resume`/`continue` when no runtime/orchestrator is bound and API chat proxy is available | `agent/telegram_bridge.py::handle_telegram_text()` | Routes to API chat proxy | Normal Telegram chat proxy result | Yes. Avoids a stale local memory fallback. | Low |
| Social/presence fast path | Trivial greetings, acknowledgements, and presence checks | `agent/public_chat.py` via `agent/telegram_bridge.py::handle_telegram_text()` | Before API chat proxy and before Orchestrator | Short user-facing text with command next action for presence checks | Yes. Keeps simple Telegram turns fast and deterministic. | Medium |
| Direct runtime/orchestrator fallback | Tests or direct callers pass a runtime or orchestrator without API proxy | `agent/telegram_bridge.py::_canonical_chat_result()` | May call `runtime.chat()` or direct `orchestrator.handle_message()` | Telegram chat result with legacy compatibility marker; text is sanitized | Yes for local tests/direct integrations only. Production Telegram should use `api_chat_proxy`. | Medium |
| Numeric no-wizard text | Telegram text like `1` or `2` with no local command/wizard state | `agent/telegram_bridge.py::handle_telegram_text()` | Routes to API chat proxy or runtime chat | Normal chat result | Yes. Numbers are not commands unless a pending chooser exists in the API/runtime. | Low |

Bypasses should stay limited to protocol validation, explicit compatibility continuations, health/setup commands, and deterministic social turns. When a path has enough user intent and the chat runtime is reachable, it should route through `AgentRuntime.chat()`.

For live validation after promoting stable runtime changes, see [../operator/LIVE_BEHAVIOR_SMOKES.md](../operator/LIVE_BEHAVIOR_SMOKES.md).
