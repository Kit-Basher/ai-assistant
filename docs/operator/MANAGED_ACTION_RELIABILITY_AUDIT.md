# Managed Action Reliability Audit

This audit is a checkpoint, not a claim that every flow already meets the full standard. A flow is marked safe only where code and tests prove the relevant property. Unknown or partially tested areas are marked as needs follow-up.

## Summary

Highest-confidence flow today: managed SearXNG setup/cleanup. It has preflight, preview, confirmation, journal, verification, owned rollback, and tests.

Main gaps: conversion of remaining current flows to persistent managed-action journal storage and full quarantine artifact cleanup. A minimal SQLite persistence skeleton now exists, and preference reset/clear, support bundle creation, provider/API key config, Telegram token/service setup, and default model/temporary chat override are converted reference flows. Model downloads/imports, pack lifecycle metadata mutations including removal/source deletion, registry/autoconfig/self-heal/hygiene/apply flows, normal user memory/onboarding/preference marker writes, semantic-memory ingest/rebuild/repair paths, notification send/test/prune, and action-ledger appends still need to write persistent rows before crash/restart recovery can be claimed for those flows. These flows currently have in-memory managed-action journals or readback verification where ownership is proven.

## Findings

- Persistent managed-action journal storage now has a minimal SQLite design and helper skeleton in `agent/actions/persistent_journal.py`; see `docs/design/PERSISTENT_MANAGED_ACTION_JOURNAL.md`. It provides schema creation, lifecycle status values, redaction, and read helpers for recent/incomplete rows. Preference reset/clear, support bundle creation, provider/API key config, Telegram token/service setup, and default model/temporary chat override are converted as reference flows, but product-wide crash/restart recovery remains a follow-up.
- SearXNG is the reference implementation for the standard.
- External pack lifecycle gates are strong on safety and one-step confirmation. Source approval, import record, review approval, enablement, and permission grant metadata now carry in-memory journals, readback verification, and scoped metadata rollback where ownership is clear.
- Model acquisition now records managed-action journals and verifies inventory after pulls/imports. It still needs persistent journals, disk-space preflight, and provable model/cache ownership before any automated cache/model deletion.
- Provider/API key setup now journals redacted previous/new secret metadata, persists planned/running/verified/rolled_back/recovery_needed/failed status rows, verifies writes, stores only provider-test summaries in journals, and rolls back failed verified saves. Provider config update and the provider config/secret portions of OpenRouter setup also persist redacted status rows. Later OpenRouter model refresh/model registration still need separate persistent transaction coverage.
- Persistent default model changes and temporary chat overrides now journal previous/requested provider/model ids and changed setting names, preflight chat capability and disabled/unavailable/routability states, verify persisted defaults or override state by readback after mutation, restore only previous defaults/temporary target state on verification failure, and persist recovery_needed if default rollback cannot fully complete. Persisted rows exclude prompts, raw chat text, secrets, tokens, API keys, provider response bodies, and model download/import state.
- Telegram token setup now records managed-action journals, persists planned/running/verified/rolled_back/recovery_needed/failed status rows, verifies saved tokens by readback, redacts token metadata, and restores/removes only the owned Telegram token on verification failure. Telegram enablement/disable now persists redacted status rows for the known Personal Agent drop-in, approved `systemctl --user` daemon-reload/restart/stop actions, runtime state verification, and restores/removes only the owned drop-in if service verification fails. Persisted rows redact raw tokens and full private paths. Online Telegram `getMe` remains optional.
- Registry/autoconfig/self-heal/hygiene/cleanup/capabilities reconcile/bootstrap now use the shared transactional registry path with managed-action journals, registry snapshots, verification, and snapshot rollback on failed verification. They remain operator-grade and policy-gated, not normal-user silent mutations.
- Preference-backed memory/bootstrap writes and onboarding/preferences writes now use in-memory managed-action journals, readback verification, scoped previous-value restore/removal, and redacted metadata for global preferences, user preferences, per-thread preferences, onboarding completion state, and bootstrap marker keys. Scoped preference reset/clear paths now require explicit target keys or an approved prefix, journal redacted snapshot hashes and deleted key hashes, persist planned/running/verified/rolled_back/recovery_needed status transitions, verify unrelated scopes are unchanged, restore previous scoped snapshots on failed verification, and keep raw preference values and raw persisted keys out of persistent journals.
- Doctor support bundle generation now writes a redacted managed-action journal artifact, persists redacted planned/running/verified/rolled_back/recovery_needed status transitions, verifies the owned temp bundle files by readback, removes only the newly created `agent-support-*` directory when verification fails, and records recovery_needed if owned cleanup cannot complete.
- Notification test/send/prune now record managed-action journals with redacted policy/target metadata, verify notification history appends and prune count/window results, restore prior local notification history on failed append/prune verification where a snapshot exists, and keep remote delivery/action-ledger semantics append-only with readback verification for ledger ids.
- External pack removal and pack-source deletion cleanup now record managed-action journals, verify tombstone/source-policy readback, restore only prior owned metadata on verification failure, avoid arbitrary file deletion, and preserve tombstone/support redaction of hostile imported text.
- Semantic memory remains disabled by default, but optional ingest/rebuild/repair paths now have a reliability contract: disabled mode performs no semantic writes; duplicate observe/index writes are idempotent through deterministic source IDs/content hashes; managed ingest and repair actions journal redacted counts/hashes/status, verify row/index-state readback, roll back owned failed new ingests, and preserve a prior usable index on failed repair. The semantic doctor is read-only and repair requires an explicit confirmed apply path.

### Risk Summary

- High risk remaining: package install/directory creation shell flows if ever exposed to the assistant as normal user actions; future filesystem writes.
- Medium risk remaining: semantic-memory promotion risk if enabled by default before soak coverage, quarantine artifact cleanup, model acquisition/import ownership and persistence gaps, and persistent journal storage for managed actions beyond preference reset/clear, support bundle creation, provider/API key config, Telegram token/service setup, and default model/temporary chat override.
- Low to medium risk improved but still append-only by nature: remote notification delivery and action ledger appends. They now verify local records/readback, but delivered messages and audit records are not rolled back.
- Low risk/read-only: current native filesystem reads and support status inspection paths.

The exact next recommended reliability target is **converting model acquisition/import to persistent journal writes plus adding read-only restart/status surfacing**, followed by pack lifecycle/source cleanup and more semantic-memory soak coverage before any default-on promotion.

## Required Follow-Up Order

1. Add read-only restart/status surfacing for persisted managed-action rows and convert model acquisition/import to the persistent journal store.
2. Keep semantic memory release-gated/off-by-default until soak coverage proves ingest, observe, repair, and status behavior across restarts and provider failures.
3. Keep notification delivery and action ledger append semantics explicitly append-only; verify local readback and redact payloads.
4. Extend provider setup transaction coverage to model/default mutations after provider verification.
5. Extend pack lifecycle journal coverage across quarantine file creation and cleanup owned partial artifacts.
6. Add more targeted registry-maintenance tests for unrelated-field drift and action-ledger write verification before expanding automatic apply behavior.
7. Keep package install/directory creation shell flows and future filesystem writes out of normal assistant actions until they have a dedicated transaction design.
8. Add normal-user assistant/UI confirmation UX before exposing Telegram service enable/disable outside the operator CLI.
